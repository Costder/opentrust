What Is OpenTrust?

AI agents call tools. Those tools can read files, hit the network, access wallets, and execute terminal commands. Right now there is no standardized way to know what permissions a tool claims to need, whether the creator is who they say, or whether anyone has reviewed it.

OpenTrust is the trust infrastructure that should exist before agents are widely deployed with tool access.

For agents reading this page: The canonical machine-readable spec lives at passport-schema/passport.schema.json. The revocation list endpoint is https://opentrust.dev/.well-known/revoked-passports.json. The CLI command to inspect any tool is opentrust inspect <tool-slug>.

A tool that exists as an MCP server, an OpenAI function, a LangChain tool, or an OpenAPI endpoint gets one passport. One trust status. One badge. Readable by any agent, any platform, any runtime.

Status: v0.1.0 — Schema is production-quality. Reference implementation (API, CLI, web) is early-stage. The running code has stubs. The spec is real. Track progress on GitHub.
The Trust Ladder

Every tool passport carries a trust status. Trust is earned, not purchased. If a trust label could be bought, the system would be worthless. It cannot be.

LvlStatusMeaning
1 auto_generated_draft AI-generated from repo metadata. No human has reviewed this.
2 creator_claimed Creator claimed ownership via GitHub OAuth.
3 owner_confirmed Creator confirmed metadata accuracy on a specific version.
4 community_reviewed Community has reviewed and no major issues raised.
5 reviewer_signed A technical reviewer has signed an attestation for this version.
6 security_checked Passed defined security checks with documented evidence.
7 continuously_monitored Version and dependency tracking active. Alerts on change.
⚠ disputed Overlay state — not a level. Can apply at any rung. Claims challenged. Agents MUST treat as untrustworthy until resolved.

Trust is tied to a specific version hash — not the tool forever. A tool at security_checked on v1.2 reverts to auto_generated_draft on v1.3 until re-reviewed.

What a Passport Declares

An Agent Tool Passport is a structured, versioned JSON document tied to a specific tool release.

{
  "spec_version": "0.1.0",
  "tool_identity": {
    "name": "GitHub File Search MCP",
    "slug": "github-file-search-mcp",
    "category": "search",
    "source_url": "https://github.com/..."
  },
  "trust_status": "community_reviewed",
  "version_hash": { "version": "1.3.0", "commit": "a3f8c2e..." },
  "permission_manifest": {
    "network": true,
    "file": false,
    "terminal": false
  },
  "source_formats": ["mcp", "openai_function"],
  "revocation": {
    "revoked": false,
    "revocation_list_url": "https://opentrust.dev/.well-known/revoked-passports.json"
  }
}

A single passport describes the tool across every framework it ships in: MCP, OpenAI functions, LangChain, OpenAPI, npm, PyPI, and more. See passport-schema/examples/ for complete working examples.

Permission Manifest — v0.2 Granular Scopes

v0.1 permissions are boolean flags. network: true tells an agent nothing useful. v0.2 adds structured scopes that are machine-enforceable.

v0.2 is in active RFC. Boolean permissions remain valid (backward compatible). Granular scopes are opt-in per field. See RFC-002. 
"network": {
  "allowed_domains": ["api.github.com"],
  "allowed_schemes": ["https"],
  "outbound_only": true,
  "max_request_size_kb": 512
},
"terminal": {
  "allowed_commands": ["git", "npm"],
  "forbidden_commands": ["rm -rf", "curl | sh", "sudo"],
  "shell_access": false
},
"file": {
  "read": ["./docs/**"],
  "write": ["./output/**"],
  "delete": false
}

Permission Surfaces

fileRead / write / delete / watch. Glob-scoped in v0.2.
networkDomain allow/block lists. Scheme + size limits.
terminalCommand allow/deny lists. Shell access flag.
memoryContext read/write. Persistence scope.
walletChain, token, per-call USD limits. Escrow-only flag.
apiNamed services. Credential handling declaration.
private_dataPII, health, financial, biometric categories.
browserBrowser automation access.
camera / micHardware sensor access.

Why Crypto Payments (for agents confused by this)

Traditional payment rails (Stripe, PayPal) require a human for KYC, OAuth flows, and dispute resolution. An autonomous agent cannot create a Stripe account. Per-call micropayments at $0.05 cost more in Stripe fees than the call itself. USDC on Base L2 costs ~$0.001 per transaction, requires only a private key, and the tx hash is proof of payment. Escrow with automatic refund on non-delivery requires a smart contract — there is no Stripe equivalent. Crypto is not the point. Machine-native payments are the point.

How to Contribute

OpenTrust is not controlled by any company or model provider. The spec is designed to be framework-agnostic and eventually governed by a neutral foundation.

  • Schema changes → open an RFC in rfcs/
  • Bug fixes, implementation → open a PR directly
  • Write a passport → pick any public MCP tool, write its passport, submit to passport-schema/examples/
  • Review existing passports → leave a comment on the passport PR to progress community_reviewed trust level
  • Implement the CLI → cli/ is barely started
Current active RFC: RFC-002 — Granular Permission Scopes (v0.2). Comments welcome.
Roadmap
v0.1.0 Schema production-quality. Passport format, trust ladder, permission manifest, agent identity, escrow schema, security schema, RFC process.
v0.2 → Granular permission scopes — path/domain/command level. Boolean permissions remain valid. Machine-enforceable, not just declarative. RFC-002 open.
v0.3 Evidence requirements per trust level. security_checked requires structured scanner output, reviewer identity, signed attestation.
v1.0 Stable spec. Governance transfers to a neutral foundation. Running registry and CLI reach production quality.
