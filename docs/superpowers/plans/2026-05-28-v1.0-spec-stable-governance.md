```markdown
## v1.0 — Stability Milestone and Governance Transition Plan

OpenTrust reached v1.0 on 2026-05-28. This marks two things:

### What "stable" means for the spec

- The passport schema at v1.0.0 will not have breaking changes without a major version bump.
- A breaking change is: removing a required field, renaming a field, changing a field's type, or tightening an enum that previously accepted a value.
- Additive changes (new optional fields, new enum values) are non-breaking and go through a standard RFC.
- The 90-day migration window rule: any breaking change must be announced at least 90 days before the old schema stops being accepted by the reference registry.

### Governance transition path

OpenTrust's governance will transfer to a neutral foundation when all three conditions are met:

1. **Schema stability:** The v1.0.0 spec has been in production use for at least 6 months with no emergency breaking changes.
2. **Multi-ecosystem adoption:** Passports are being issued and consumed by tools in at least 3 distinct agent ecosystems (e.g. MCP, LangChain, OpenAI tools).
3. **Community reviewer pool:** At least 5 independent reviewers have signed attestations in the registry using their own registered keys (not the registry's key).

Until those conditions are met, the spec is maintained by [@Costder](https://github.com/Costder) with the RFC process as the primary governance mechanism.

### What changes at governance transfer

- The canonical schema repository moves to a neutral GitHub org (e.g. `opentrust-protocol`).
- The spec maintainer role becomes a rotating elected position (annual term, community vote).
- The registry signing key is moved to a hardware security module (HSM) controlled by the foundation.
- OpenTrust's financial neutrality clause becomes a binding charter rule, not just a README statement.

### What does NOT change at governance transfer

- The MIT license stays.
- The payment neutrality rule (no fee, no treasury, no cut) stays.
- The RFC process stays — it becomes the foundation's primary governance mechanism.
- All existing passports remain valid — no re-issuance required.

---

*This section was added at v1.0.0 (2026-05-28). It will be updated when the transfer happens.*
```

Create `CHANGELOG.md` at the repo root:

```markdown
# Changelog

All notable changes to OpenTrust are documented here.
Format: [Semantic Versioning](https://semver.org). Dates: ISO 8601 (YYYY-MM-DD).

---

## [1.0.0] — 2026-05-28

### Summary

First stable release. Passport schema is frozen. All four milestone features from the v0.x roadmap are complete.

### Added — v0.2: Granular permission scopes

- `permissions.schema.json` defines structured scope objects for `file`, `network`, `terminal`, `wallet`, `memory`, `api`, and `private_data` permissions.
- `reviewer_signed` and above passports must use granular scopes (structured objects) for high-risk permission surfaces — boolean `true` is rejected at these trust levels.
- `manifest-validator/validator.py` upgraded to use the full schema bundle including `security.schema.json`.
- CLI `validate` command enforces granular scope requirement and reports helpful error messages.
- API `POST /tools` and `PUT /tools/{slug}` enforce granular scopes at `reviewer_signed+`.
- Boolean `true` remains valid for `creator_claimed` and below (backward compat).

### Added — v0.3: Evidence requirements

- `SecurityEvidenceBlock` added to passport schema: required fields `scanner_output`, `reviewer_identity`, `commit_hash`, `dependency_snapshot`, `signed_attestation`.
- `security_checked` and `continuously_monitored` passports must include a complete `SecurityEvidenceBlock`.
- CLI and API enforce the requirement; partial evidence blocks are rejected with field-level errors.

### Added — v0.6: Real marketplace flows

- On-chain USDC payment verification on Base L2 via `web3.py` (reads transaction receipts, validates amount/sender/recipient).
- Escrow order flow: `POST /marketplace/orders` with a `transaction_hash` verifies the payment on-chain before creating the order.
- Embedded wallet generation: `POST /wallets/generate` creates a new EVM wallet server-side, encrypted with AES-256-GCM per-user key. Requires `OPENTRUST_EMBEDDED_WALLET_ENABLED=true`.
- New endpoint: `POST /payments/verify-onchain` — standalone USDC transfer verification for any tx hash.
- Feature flags: `BASE_RPC_URL`, `BASE_USDC_CONTRACT`, `WALLET_ENCRYPTION_SECRET` added to config.

### Added — Hands and Feet v0.2.0 (`@opentrust/hands-and-feet`)

- `prepare_payment` composite MCP tool: detects Base USDC balance, bridges Polygon→Base via Across Protocol if needed, polls until minted, executes `pay_with_usdc`. Returns full receipt with bridge fee.
- Bridge timeout (30 min), stuck detection, and clear error messages with recovery URL.
- Total cost (amount + bridge fee) returned in receipt.

### Changed

- All package versions bumped to `1.0.0` (api, cli, sdk, payment-contracts, hands-and-feet).
- Passport schema `spec_version` description updated to `1.0.0` with stability guarantee.
- All example passports updated to `spec_version: "1.0.0"`.
- `docs/governance.md` updated with v1.0 stability guarantees and governance transfer plan.

---

## [0.5.0] — (prior)

- Spend policy + signed payment quotes. Deny-first local spend policy, signed and expiring payment quotes, nonce protection against replay, wallet-bound quotes, escrow threshold enforcement.

## [0.4.0] — (prior)

- Signed registry + revocation. Ed25519 signing on all passports, pinned public keys at `/.well-known/opentrust-keys.json`, signed revocation list with monotonic versioning and rollback rejection, offline CLI verification.

## [0.1.0] — (initial)

- Reference registry. Passport schema, FastAPI CRUD, CLI, Next.js frontend, badge generator, Docker Compose, CI.
- `@opentrust/hands-and-feet` v0.1.0: Full V1–V3 MCP capability layer (~50 tools). See packages/hands-and-feet/README.md.
```
